# Performance Audit — Raw Findings

Audit target: OpenEMR codebase at `/Users/macbook/dev/Gauntlet/week1/openemragent`. Scope: latency the AgentForge clinical co-pilot will inherit when it calls internal REST/FHIR APIs to fetch patient context.

Severity legend: **CRITICAL** = makes the agent feel broken (>5 s p95). **HIGH** = ~hundreds of ms hotspot. **MEDIUM** = scale concern (gets worse over months/years of data). **LOW** = nit.

## Summary (biggest agent-latency risks)

- **Bootstrap cost is enormous and runs on every API call.** `interface/globals.php` is `require_once`'d on every REST/FHIR request from `SiteSetupListener`, and runs `SHOW TABLES`, fetches all rows from `globals` (DB-backed key/value store), fetches user-specific overrides from `user_settings`, instantiates a Symfony `Kernel` + `ModulesApplication`, then logs the HTTP request to the audit chain. This is a fixed ~50–150 ms tax per agent tool call before the first byte of work. (`/Users/macbook/dev/Gauntlet/week1/openemragent/interface/globals.php:398-509`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/RestControllers/Subscriber/SiteSetupListener.php:202`).
- **Every API request writes to 2–3 audit tables and recomputes a sha3-512 checksum.** `LogTablesSink` does `INSERT INTO log` + `INSERT INTO log_comment_encrypt` + (for API calls) `INSERT INTO api_log`, with a 64-byte sha3-512 hash over the full row. The full JSON response body is captured into `api_log.response` (longtext) — so if the agent fetches a 500-resource Bundle, the 500-resource JSON is also persisted in `api_log` as a side effect. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Common/Logging/Audit/LogTablesSink.php:38-100`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/RestControllers/Subscriber/ApiResponseLoggerListener.php:71-93`).
- **Audit-on-every-SELECT compounds it.** `QueryUtils::sqlStatementThrowException` calls `Execute()` (audited) by default; if the query touches a table in `LOG_TABLES` (patient_data, lists, prescriptions, immunizations, form_encounter, history_data, pnotes, billing, …) `auditSQLEvent` writes another row. So one tool call that reads, e.g., problems + meds + labs can produce 6–10 audit rows. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Common/Database/QueryUtils.php:209-223`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Common/Logging/EventAuditLogger.php:116-150,390-509`).
- **No application-level cache.** The repo declares `symfony/cache: ^7.3` in `composer.json` but no service uses it. `list_options` (~2.4 k seed rows, often more) is re-fetched per FHIR resource per request via `ListService::getOptionsByListName` which is `SELECT * FROM list_options WHERE list_id = ?` — callers cache only on the per-instance object, and a fresh service is constructed per request. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/ListService.php:121-146`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/FHIR/FhirPatientService.php:111-116,534-549`).
- **FHIR services build the result with giant subquery-soup queries that materialize whole tables.** Patient, Encounter, Allergy, Condition, Immunization, Prescription, Procedure, and Vitals services all use the pattern `LEFT JOIN (SELECT ... FROM <table>) alias ON ...` for nearly every joined table. MySQL is reasonable at flattening these but it cannot push the outer WHERE pid filter into every subquery; for a single-patient lookup the planner often materializes full scans of `patient_data`, `users`, `facility`, etc. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/EncounterService.php:187-315`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/ProcedureService.php:81-289`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/AllergyIntoleranceService.php:44-98`).
- **Classic N+1 in `ProcedureService::getAll`.** Per `procedure_order` row, three follow-up queries: `SELECT … FROM procedure_order_code`, `SELECT … FROM procedure_report`, then per `procedure_report` another `SELECT … FROM procedure_result`. A patient with 50 orders × 3 reports = 50 + 50 + 150 = ~250 queries to satisfy one ServiceRequest read. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/ProcedureService.php:701-754`).
- **Service constructors call `UuidRegistry::createMissingUuidsForTables` on every instantiation.** This runs `SELECT count(*) … WHERE uuid IS NULL` per table on every controller hit. On a freshly imported / restored DB the same call will then back-fill UUIDs in a transactional loop *inside* the agent's API request, blocking the response for tens of seconds. (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/EncounterService.php:64-69`, `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Common/Uuid/UuidRegistry.php:295-438`).
- **CCDA generation hops to a Node service over a TCP socket and may `sleep(5)` on cold start.** If the agent ever needs to ask for a CCDA, the PHP side opens a socket to `127.0.0.1:6661`, and on a cold start does `exec("node serveccda.js ... &"); sleep(5);` synchronously. (`/Users/macbook/dev/Gauntlet/week1/openemragent/interface/modules/zend_modules/module/Carecoordination/src/Carecoordination/Model/CcdaServiceDocumentRequestor.php:46-78`).

## Database Scale and Shape

The agent will mostly read these tables. Sizes are based on schema (`/Users/macbook/dev/Gauntlet/week1/openemragent/sql/database.sql`, 15 395 lines, 282 tables) and column counts; row-count expectations assume a "real" practice (≥10 k patients with 5 yrs of history).

- **`patient_data`** — `:8334-8472`. ~110 columns including dozens of `varchar(255)`, several `longtext` (`occupation`, `industry`, etc.), 25+ `usertext*`/`userlist*` slots. `SELECT * FROM patient_data WHERE pid = ?` returns ~3–5 KB / row. Indexes: `UNIQUE pid`, `UNIQUE uuid`, `idx_patient_name(lname, fname)`, `idx_patient_dob(DOB)`, `KEY id`. **Note `pid` is unique-keyed, not the PK — `id` is PK and is also separately keyed which is redundant.** Row count expectation: 10 k–500 k.
- **`form_encounter`** — `:2022-2062`. ~30 columns. Indexed by `UNIQUE uuid`, `pid_encounter(pid, encounter)`, `encounter_date(date)`. The `pid_encounter` composite is good for `WHERE pid = ? ORDER BY encounter`, but most FHIR queries `ORDER BY date`/`ORDER BY eid DESC`. Row count: 100 k–10 M (encounters dominate volume).
- **`forms`** — `:2460-2478`. Light table tracking which form rows belong to which encounter. Indexes `pid_encounter(pid,encounter)`, `form_id(form_id)`. Note `formdir` is `longtext` and **not indexed** — `getFormNameByFormdir` and `getFormIdByFormdirAndFormid` (`/Users/macbook/dev/Gauntlet/week1/openemragent/library/forms.inc.php:122-131`) query on `formdir`, forcing a full scan.
- **`history_data`** — `:2916-3082`. ~150+ columns, mostly `longtext` and `varchar(255)`. Wide PHI table. Indexes: `KEY pid` and `UNIQUE uuid`. No date index; `getHistoryData` does `ORDER BY date DESC, id DESC LIMIT 1` — uses pid index then filesort. Row count ~ 1 row/patient (sometimes more).
- **`lists`** — `:7671-7712`. Stores problems/medications/allergies in one table discriminated by `type`. Indexes: `KEY pid`, `KEY type`, `UNIQUE uuid`. **No composite `(pid, type)`** so `WHERE pid = ? AND type = 'medical_problem'` uses one index and filters; tolerable at small scale, table-scanning-y at million rows.
- **`prescriptions`** — `:8697-8751`. ~50 columns. Indexes: `KEY patient_id(patient_id)`, `UNIQUE uuid`. `drug` is varchar(150), `note` and `drug_info_erx` are TEXT. No index on `(patient_id, active)` so listing only active meds for the agent does a per-row filter.
- **`procedure_result`** — `:10493-10513`. **Has no `patient_id` column.** Joined indirectly through `procedure_report` → `procedure_order` to find the patient. Indexes: PK on `procedure_result_id`, `KEY procedure_report_id`, `UNIQUE uuid`. Lab results for one patient require a 3-table join chain.
- **`procedure_order`** — `:10369-10414`. Indexes: `UNIQUE uuid`, `KEY datepid(date_ordered, patient_id)`, `KEY patient_id`, `idx_specimen_type`, `idx_scheduled_date`, `idx_order_intent`, `idx_location_id`. Decent.
- **`immunizations`** — `:3235-3287`. Indexes: `KEY patient_id`, `UNIQUE uuid`. `cvx_code varchar(64)` joined to `codes.code` (codes table can be ~100 k+ rows).
- **`drugs`** — `:1599-1626`. Indexes: PK `drug_id`, `UNIQUE uuid`. **No index on `name`, `ndc_number`, or `drug_code`** — searching by RxNorm/NDC requires a full scan.
- **`pnotes`** — `:8669-8689`. Indexes: only `KEY pid`. Body is `longtext` — pulling recent pnotes for a patient transports the entire body field across the wire.
- **`billing`** — `:245-278`. Indexes: only `KEY pid`. **No index on `encounter`** even though many queries filter by encounter.
- **`insurance_data`** — `:3306-3344`. Indexes: `UNIQUE uuid`, `UNIQUE pid_type_date(pid, type, date)`. Good.
- **`documents`** — `:1391-1432`. Indexes: PK, `UNIQUE drive_uuid`, `UNIQUE uuid`, `KEY revision`, `KEY foreign_id`, `KEY foreign_reference(foreign_reference_id, foreign_reference_table)`, `KEY owner`. `foreign_id` is the patient pid for patient-attached docs. `document_data MEDIUMTEXT` can hold inline blobs.
- **`list_options`** — `:3887-3905`. PK `(list_id, option_id)`. Seed data inserts ~2 377 rows (`grep -c "INSERT INTO list_options"`). Read constantly to translate option_id → display string for FHIR resources, and the SELECT is `SELECT *` (returns `notes TEXT`, `last_updated`, `timestamp`, etc.).
- **`codes`** — `:1124-1144`. Indexes: PK `id`, `KEY code`, `KEY code_type`. With ICD-10 / RxNorm / SNOMED imported via `contrib/`, this table can grow to **hundreds of thousands of rows**. Joins from `immunizations.cvx_code = codes.code` without `code_type` filter will pick the wrong codes.

## Query Patterns (N+1, missing LIMIT, etc.)

- **[CRITICAL] `ProcedureService::getAll` is N+1+M.** `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/ProcedureService.php:701-754`. Per `procedure_order` row in the outer result set: `SELECT … FROM procedure_order_code WHERE procedure_order_id = ?`, `SELECT … FROM procedure_report WHERE procedure_order_id = ?`, then per `procedure_report` row `SELECT … FROM procedure_result WHERE procedure_report_id = ?`. For an oncology / chronic patient with 100+ orders this hits ~400+ queries.

Each Patient resource read hits `list_options` 4–6 times (race, ethnicity, language, gender_identity, …) with `SELECT * FROM list_options WHERE list_id = ?`.
- **[HIGH] `BaseService::addCoding` is N+1 and runs `SHOW TABLES` per code.** `/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/BaseService.php:551-573` calls `lookup_code_descriptions` which (`/Users/macbook/dev/Gauntlet/week1/openemragent/custom/code_types.inc.php:776-870`) explodes the `;`-separated codes and per-code does **`SHOW TABLES LIKE '<join_table>'`** then a SELECT against the code table. ConditionService and ProcedureService call `addCoding` once per row in the result loop.
- **[HIGH] Patient search runs three sequential queries plus a 5-way LEFT JOIN that materializes subqueries.** `PatientService::search` issues a count, then a uuid-list query, then a third query that re-joins everything against the uuid IN list (`/Users/macbook/dev/Gauntlet/week1/openemragent/src/Services/PatientService.php:418-523`). The base join includes `LEFT JOIN (SELECT … FROM patient_history)`, `LEFT JOIN (SELECT … FROM users)`, `LEFT JOIN (SELECT contact … contact_address … addresses … WHERE contact.foreign_table_name='patient_data')`. `contact.foreign_table_name` is **not indexed** (`:1167-1173`), so the subquery scans contact + contact_address + addresses every time.

## CCDA / FHIR / External Service Hops

- **[CRITICAL] CCDA generation is a TCP socket round-trip to a Node service that may not be running.** `/Users/macbook/dev/Gauntlet/week1/openemragent/interface/modules/zend_modules/module/Carecoordination/src/Carecoordination/Model/CcdaServiceDocumentRequestor.php:46-89`. PHP opens a `socket_create(AF_INET, SOCK_STREAM)` to `127.0.0.1:6661`. **If the node service is not already up:**
  - `exec("$command $path/serveccda.js > /dev/null &")`
  - `sleep(5)` — synchronous five-second sleep blocking the response
  - then attempts `socket_connect` again.
- **[HIGH] CCDA write loop has `usleep(200000)` per chunk.** `:103-113` of the same file, `do { socket_write(...); usleep(200000); } while ($pos < $len);`. For a CCDA document with 200 KB of XML and a 64 KB write buffer, that's at least 4 chunks × 200 ms = 800 ms baked-in latency, regardless of document complexity.

## Estimated Latency Budget for the Agent (back-of-envelope only)

Assuming Docker dev image (default php.ini, opcache off), MySQL on the same host, audit logging on (default), encryption off (default), patient with ~100 encounters / 30 active meds / 50 lab results.

| Phase | Cost per agent tool call |
|---|---|
| `dispatch.php` cold autoload + `vendor/autoload.php` (no opcache) | 30–80 ms |
| `globals.php` (SHOW TABLES + globals + user_settings + timezone + Kernel + Modules + log HTTP request + xlWarmCache) | 30–100 ms |
| Session start (file-backed) | 5–20 ms |
| ACL `aclCheckCore` (recursive admin/super check + `acl_query` joins) | 5–15 ms |
| Service constructor `UuidRegistry::createMissingUuidsForTables` (3–4 count queries) | 10–30 ms |
| One FHIR Patient read (3 query passes + 4–7 list_options selects + N audit inserts) | 50–150 ms |
| One FHIR Encounter search by patient (no LIMIT, ~100 encounters × giant LEFT JOIN soup + audit) | 100–600 ms |
| One FHIR Condition / Allergy search by patient (RIGHT JOIN entire patient_data) | 100–500 ms |
| One FHIR MedicationRequest list (UNION prescriptions+lists, 8 list_options joins, FhirOrganization per row) | 150–800 ms |
| One FHIR Procedure / Lab list (N+1+M loop) | 300 ms – 5 s+ |
| `ApiResponseLoggerListener::onRequestTerminated` (encrypt + hash + 3 inserts) on a 500-row bundle | 100–500 ms |
| Cold-start CCDA generation (sleep(5) + node spawn + socket usleep loop) | 5 000–10 000 ms |

A "minimal" agent prompt that fetches Patient + 5 active resources for context = roughly **600 ms – 3 s** in the warm path, **multi-second to tens of seconds** if any of these hit cold/N+1 paths or background services kick in. CCDA-derived context is effectively unusable for synchronous agent flow.

- **Symfony Twig compile-cache state.** Twig templates compile on first render; not directly on the FHIR JSON path so probably not blocking the agent, but did not measure.
